# Anticon slide review ritual

A structured walk through the Anticon deck, one slide at a time. We do NOT skip ahead, we do NOT batch multiple slides into one pass. One slide, four steps, then wait for feedback.

All paths in this skill are relative to the repo root.

## Active artifacts

- Deck: `Drafts/anticon-talk-draft-007.html`
- Screenshots: `Drafts/screenshots-draft-007/NN-slide-N.png` (00–17, where 17 is `slide-reveal`). Generated by `Drafts/capture-deck.js`; not committed to the repo.
- **Authored intent per slide:** `Research/presentation-working-notes.md`, sections titled `## Slide N — …`. **This is the canonical "what we're trying to say"; load before reviewing.**
- Talk arc / outline: `Research/01-talk-outline-from-builder.md` (short, ~70 lines).
- Quotes & data points: `Research/03-key-quotes-and-data-points.md`.
- Constraints + open issues: `Drafts/CLAUDE.md`
- Hard-won lessons from earlier sessions: `Drafts/docs/gotchas.md`

If the deck has been edited since the screenshots were captured, re-run `node capture-deck.js` from the Drafts directory before continuing.

## Before reviewing a slide, load the authored intent

**Always load the intent before writing the review.** Don't reverse-engineer
intent from what's rendered, the authored notes are the source of truth.

For slide N:
1. Grep `Research/presentation-working-notes.md` for the heading
   `## Slide N — …` and read the full section that follows (it continues
   until the next `## Slide` heading or the next major `##` heading).
2. Skim `Research/01-talk-outline-from-builder.md` if you need broader arc
   context (it's short, ~70 lines).
3. Cross-reference `Research/03-key-quotes-and-data-points.md` if the slide
   carries a verbatim quote or statistic.

If `presentation-working-notes.md` has no `## Slide N` section, say so
plainly in the Intent section ("no authored intent in the working notes,
inferring from on-slide content") instead of pretending you know.

## Per-slide structure (do all four, in order, in a single message)

For each slide, output four labelled sections, no preamble, no recap of the ritual:

### 1. Content summary
Read the slide's inner HTML from the deck file. In 2–5 short bullets, say what is literally on the slide (headline, body text, callouts, citations, asset). Quote the key phrases verbatim where they carry weight. Do NOT editorialise here.

### 2. Intent (sourced from working notes)
Lead with the **authored intent from `presentation-working-notes.md`**, not
your interpretation. Quote the verbatim language the author wants to use
when it exists ("I fell in love with MarTech", "code is cheap, requirements
are expensive", etc.). Then 1–2 sentences placing the slide in the arc
(problem framing → diagnosis → way of working → reveal) and naming the
handoff to the next slide.

If the rendered slide doesn't match the authored intent (missing key
phrase, wrong emphasis, content drifted), flag the gap here, not just in
the design section.

### 3. Design suggestions
2–4 bullets, each concrete and actionable. Reference what's visible in the screenshot. Call out:
- **Intent-to-design mismatches**, does the layout surface the line the speaker actually wants to deliver?
- Layout problems (overflow, empty zones, collisions, cross-check the "Known open issues" list in `CLAUDE.md`).
- Typography or hierarchy issues (wrong font role, competing weights, headline buried).
- Composition (where the eye lands first vs. where it should land).
- Anything the hand-drawn / notebook direction is doing well or fighting against.

Stay within the **locked constraints** in `CLAUDE.md` (palette, fontstack, spatial coordinates, impress.js version) unless the user explicitly opens those up.

### 4. Awaiting feedback
End with a single line inviting content + design feedback for THIS slide. Do not move on.

## Movement rules

- Start on slide 0 unless the user names a different slide.
- After the user gives feedback, apply it (edit the deck) OR record it for later, whichever they ask. Then ask before moving to the next slide.
- Never preview slide N+1 in the same message as slide N's review.
- If the user redirects ("skip", "jump to slide 7", "let's restart on this one"), follow them, don't fight the order.
- When all slides reviewed, summarise the cross-cutting themes that emerged and propose a design-finalisation plan.

## After every code change, design critique by agent

**Before reporting a slide as done, run an agent critique on the rendered screenshot.**
Self-review is unreliable; spawn a fresh Agent (`general-purpose`) with the
screenshot path and ask it to honestly critique formatting, positioning,
alignment, spacing, hierarchy, and any obvious visual problems. The agent
hasn't been arguing with the design, it sees the rendered image cold.

Workflow:
1. Make the edit.
2. Re-run `capture-deck.js` (or `screenshot-slide0.js` for slide 0 alone)
   so the screenshot reflects the change.
3. Launch an Agent with this brief:

   > Look at the screenshot at `<path>`. This is slide N of an
   > impress.js deck rendered at 1920×1080. The intent of this slide is
   > `<one-line intent from working notes>`. Be brutally honest about the
   > formatting: alignment problems, overlapping elements, awkward empty
   > zones, weak hierarchy, text crashing into other elements, things that
   > look like bugs rather than design choices. Don't be polite. Report
   > 3–6 specific issues with pixel-ish locations, plus one or two things
   > that are actually working.

4. Apply the agent's feedback **as signal, not as directive**. Agent
   suggestions are an outside read, not orders. If a suggested fix would
   regress something else, or contradicts a prior user decision, push
   back instead of blindly applying. (Real case: an agent recommended
   pushing the slide-1 image tilt to "~1.5–2deg" with a deeper shadow,
   applying that verbatim made the image look amateur and the user had
   to ask for a revert. Subtle moves usually win.) Re-screenshot after
   each change.
5. Re-launch a critique if the change was non-trivial. Two critiques
   that contradict each other = stop and ask the user which direction.
6. Only after the agent's critique returns "nothing notable" (or only
   minor nits) AND your own eye on the screenshot agrees, report the
   slide as done to the user.

Never skip step 6's own-eye check. Self-assessment of layouts has
historically been the single biggest source of regressions in this deck,
but blind agent-following is the second.

## What NOT to do

- Don't dump all slides at once.
- Don't restate the ritual at the top of every slide, just produce the four sections.
- Don't propose rewriting locked constraints unless asked.
- Don't claim a design fix is done without re-running `capture-deck.js` and re-checking the screenshot.
